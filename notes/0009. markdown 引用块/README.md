## 2. 🫧 评价

这一节介绍的 markdown 引用块的语法，个人在书写笔记的时候很少会使用，就是单纯觉得不好用。

个人更习惯的做法：如果有引用的文献啥的，直接在笔记的结尾插入一个 `🔗 引用` 部分来做一个统一的汇总说明即可。

个人论点：

- 如果用到引用块，通常就是表示这部分内容是第三方的描述，而不是自己所写的内容，并且更加规范的做法是在引用内容后备注上引用来源，总觉得这样书写的成本太高了。有可能你引用的就一句话，而引用说明的描述就占了大头，字数有可能比你引用的那句话还长…… 是不是得不偿失了！
- 通常在书写某一篇笔记的时候，被引用的内容会在多个标题中出现，那是不是每处引用都要加 `>` 语法来标注一下呢？虽然规范，但是写起来很麻烦吧，还很冗余！
- 有时候明明是在写正文内容，可是突然要引入一段第三方的描述，这时候为了使用引用块，就得新起一行来书写 `>` 引用内容（因为引用块没法直接在句中使用，`>` 必须出现在段落开头），这就更麻烦了，我一句话写到一半，突然让我新起一行来写引用内容，这样的写作体验就很不好了。
- 有些 markdown 解析器会给引用块加上一些特殊的样式，从视觉上来看，会有突显的效果，因此有些“自创语义”的作者会利用这一点，刻意将一些重点强调的内容包裹到引用块中，实际上这并非“引用”语义。但是，这么做不规范，并不推荐。

## 3. 🤔 如何在 markdown 中创建引用块？

在段落开头使用 `>` 加空格：

```
> 这是一段引用内容。
```

`>` 符号后的空格不是强制要求，但加上空格更符合规范且可读性更好。

## 4. 🤔 如何创建多行引用块？

方式一：每行都以 `>` 开头：

```
> 第一行引用内容。
> 第二行引用内容。
> 第三行引用内容。
```

![img](https://cdn.jsdelivr.net/gh/tnotesjs/imgs-2026@main/2026-03-05-07-09-40.png)

方式二：只在段落首行加 `>`，其余行直接续写（懒惰写法，部分解析器支持）：

```
> 第一行引用内容。
第二行引用内容（懒惰续行）。
```

![img](https://cdn.jsdelivr.net/gh/tnotesjs/imgs-2026@main/2026-03-05-07-10-06.png)

推荐每行都添加 `>`，以保证在所有解析器中表现一致。

## 5. 🤔 如何创建嵌套引用块？

使用多个 `>` 符号实现嵌套，每增加一层嵌套就多加一个 `>`：

```
> 外层引用内容。
>
> > 内层嵌套引用内容。
> >
> > > 更深一层的引用。
```

![img](https://cdn.jsdelivr.net/gh/tnotesjs/imgs-2026@main/2026-03-05-07-10-50.png)

嵌套引用常用于表示回复或对话场景（如邮件引用）。

## 6. 🤔 引用块中可以使用其他 markdown 语法吗？

可以。引用块内部支持大多数 markdown 语法，包括标题、列表、加粗、代码等：

```
> ## 引用中的标题
>
> - 列表项一
> - 列表项二
>
> 这里有**加粗**和 `行内代码`。
```

![img](https://cdn.jsdelivr.net/gh/tnotesjs/imgs-2026@main/2026-03-05-07-11-48.png)

利用这一特性可以在引用块中构建结构完整的内容块。

## 7. 🤔 引用块前后需要空行吗？

引用块前后建议各留一个空行，以确保解析器将其正确识别为独立的引用块，而不是与前后段落合并：

```
普通段落内容。

> 引用块内容。

继续普通段落内容。
```

![img](https://cdn.jsdelivr.net/gh/tnotesjs/imgs-2026@main/2026-03-05-07-12-38.png)

连续的引用块行（即使中间没有空行）通常会被解析为同一个引用块。若要分割为两个独立引用块，需在两者之间插入一个非引用行（如空行或普通段落）。

## 8. 🤔 如何在引用块中标注引用来源？

markdown 本身没有专门的引用来源语法，通常的做法是在引用块内容末尾用 `—` 加来源信息标注，或在引用块外紧接一行说明文字：

```
> 我们所知道的一切，都只是可能性。
> — 理查德·费曼

> Stay hungry, stay foolish.

— 史蒂夫·乔布斯，斯坦福大学毕业典礼演讲，2005
```

![img](https://cdn.jsdelivr.net/gh/tnotesjs/imgs-2026@main/2026-03-05-07-17-53.png)

具体排版风格因平台和个人习惯而异，保持一致即可。
